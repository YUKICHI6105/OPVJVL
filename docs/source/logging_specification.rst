==============================
Logging Specification & Configuration
==============================

本ソフトウェアでは、操作履歴の確認、ハードウェアトラブルの調査、および測定値のデバッグを
容易にするため、EQEプロジェクトの設計を踏襲した詳細なマルチレベルロギングシステムを
採用しています。

----

.. _opvjvl-logging-file-locations:

1. ログと設定ファイルの保存先
----------------------------

実行環境（通常起動とビルド後のEXE起動）に応じて、保存フォルダの絶対パスが自動的に
切り替わります（Windows推奨規格準拠）。

通常の Python スクリプト実行時（開発モード）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

開発の利便性のため、すべてプロジェクトルート（``OPVJVL/``）内に保存されます。

* **設定ファイル (settings.json)**:
  ``[プロジェクトのルート]/settings.json``

* **ログファイル (opvjvl_software.log)**:
  ``[プロジェクトのルート]/log/opvjvl_software.log``

パッケージングされた EXE 実行時（本番モード）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

PyInstaller等でEXE化して配置された場合、書き込み権限エラーを防ぎ、Windowsの
マルチユーザー機能に対応するため、ユーザー別フォルダ（AppData）にそれぞれ隔離して
保存されます。

* **設定ファイル (settings.json)**:
  ユーザーの設定を引き継ぐため、移動プロファイル対応の **Roaming AppData** に
  書き出されます。

  ``C:\Users\<ユーザー名>\AppData\Roaming\OPVJVL\settings.json``
  （または ``%APPDATA%\OPVJVL\settings.json``）

* **ログファイル (opvjvl_software.log)**:
  PC固有のデータであり、ネットワーク同期時の負荷を抑えるため、**Local AppData** の
  ログフォルダに保存されます。

  ``C:\Users\<ユーザー名>\AppData\Local\OPVJVL\log\opvjvl_software.log``
  （または ``%LOCALAPPDATA%\OPVJVL\log\opvjvl_software.log``）

.. note::

   環境変数 ``OPVJVL_SETTINGS_PATH`` / ``OPVJVL_LOG_DIR`` を設定すると、テストや
   特殊環境向けに設定ファイル・ログ出力先を任意のパスへ差し替えられます
   （``utils.paths`` 参照）。

ローテーションと永続化
^^^^^^^^^^^^^^^^^^^^^^

``TimedRotatingFileHandler`` により、毎日深夜0時にログファイルが自動ローテーション
されます。

* 前日のログは ``opvjvl_software_YYYY-MM-DD.log`` にリネーム保存されます。
* 容量の圧迫を防ぐため、過去30日分のみを保持し、それより古いファイルは自動で
  削除されます。

出力フォーマット
^^^^^^^^^^^^^^^^

ログの出力フォーマットは以下のように統一されています。

.. code-block:: text

   YYYY-MM-DD HH:MM:SS,mmm [ログレベル] ロガー名: メッセージ

* *例（測定開始時）*:
  ``2026-07-15 10:20:05,123 [INFO] opv_viewmodel: 測定を開始しました(sample=Cell01)``

* *例（機器接続処理時）*:
  ``2026-07-15 10:20:03,884 [DEBUG2] opv_viewmodel: Keithley2400へ接続開始: COM5``

.. _opvjvl-logging-levels:

2. ログレベル設計
----------------

標準の ``INFO``、``WARNING``、``ERROR`` に加え、詳細なデバッグ情報を3段階に
分類して記録しています（OpenSSHスタイル）。

.. list-table:: ログレベルと用途
   :widths: 15 10 45 30
   :header-rows: 1

   * - レベル名
     - 数値
     - 用途と記録内容の基準
     - 主な出力例
   * - **CRITICAL**
     - 50
     - 未処理例外によるクラッシュ（``sys.excepthook`` による自己検知）
     - 「Unhandled Exception (Application Crash)」
   * - **ERROR**
     - 40
     - 測定の続行が不可能な致命的エラー。機器接続の失敗やバリデーションエラーなど。
     - 「機器接続エラー」「データの保存に失敗しました」
   * - **WARNING**
     - 30
     - 処理は続行可能だが、予期せぬ動作や設定復元の失敗などが発生した警告。
     - 「設定キー xxx の復元に失敗しました」
   * - **INFO**
     - 20
     - アプリケーションの通常動作における主要なイベントやユーザーのアクション。
     - 「測定を開始しました」「測定パラメータをsettings.jsonへ保存しました」
   * - **DEBUG1**
     - 10
     - 各処理ブロックの開始、スレッド起動パラメータなど。
     - 標準の ``logger.debug()`` はこのレベルにマッピングされる
   * - **DEBUG2**
     - 9
     - ハードウェア接続プロセスの開始、保存ファイルの決定など。
     - 「Keithley2400へ接続開始」「保存先ディレクトリの決定」
   * - **DEBUG3**
     - 8
     - 最も詳細なハードウェアとの生コマンド通信、測定ループなど。
     - 「SMUへの書き込みコマンド」「シリアル読み取り生データ」

.. _opvjvl-logging-per-module:

3. モジュールごとのロギング詳細
------------------------------

各モジュールで出力される主なログレベルの基準は以下の通りです。

① main_window ロガー (GUI制御部)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

* **INFO**: グラフ表示設定の更新、終了時の測定パラメータ保存。

* **WARNING**: 設定値の復元失敗、終了時の安全停止失敗（続行はする）。

② 各ViewModel（opv_viewmodel / jvl_viewmodel / dual_channel_viewmodel）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

* **INFO**: 測定の開始・正常終了・中断完了、CSV保存パス。

* **DEBUG1**: バリデーション結果、Worker起動パラメータ。

* **DEBUG2**: 機器への接続開始・切断処理。

③ crash_handler ロガー (app.py)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

* **INFO**: アプリケーション起動完了（使用中のQtバインディング名を含む）、
  SIGINT受信時の終了処理開始。

* **CRITICAL**: 未処理例外によるアプリケーションクラッシュ（スタックトレース付き）。

.. _opvjvl-logging-customize:

4. ログ出力のカスタマイズ
--------------------------

ログレベル（出力の閾値）は、ソースコードを書き換えることなく、設定ファイル
（``settings.json``）から動的にカスタマイズ・永続化できます。

設定ファイルによる変更
^^^^^^^^^^^^^^^^^^^^^^

``settings.json`` に以下のキーを追加・変更することで、起動時のデフォルト出力制限
レベルを制御できます。

.. code-block:: json

   {
     "console_log_level": "INFO",
     "file_log_level": "DEBUG3"
   }

* **console_log_level**: 標準出力に流れるログの最小レベル（デフォルトは ``INFO``）。
* **file_log_level**: ログファイル（``opvjvl_software.log``）に記録する最小レベル
  （デフォルトは ``DEBUG3``）。

指定可能なレベル名：``DEBUG3``、``DEBUG2``、``DEBUG1``、``DEBUG``、``INFO``、
``WARNING``、``WARN``、``ERROR``、``CRITICAL``（``utils.logger.LEVEL_MAP`` 参照）。

実行時のレベル変更
^^^^^^^^^^^^^^^^^^

``utils.logger.update_log_levels(console_level_name, file_level_name)`` を呼び出すと、
アプリケーションを再起動せずに実行中のログレベルを動的に変更できます。

.. _opvjvl-logging-crash-detection:

5. クラッシュ（未処理の例外）の自己検知
--------------------------------------

本ソフトウェアでは、キャッチされなかった未処理の例外（アプリケーションのクラッシュ）が
発生した際、自動的にスタックトレースを記録する自己検知システムを備えています。

* **検知の仕組み**:
  ``sys.excepthook`` をオーバーライドし（``app.py`` の ``log_uncaught_exception``）、
  メインスレッドおよびGUIイベントループ内で発生したあらゆる未処理例外を捕捉します。
  ``KeyboardInterrupt``（Ctrl+C）は対象外とし、標準の挙動に委譲します。

* **ログ記録**:
  ``crash_handler`` ロガーにより、**CRITICAL** レベル（数値: 50）でスタックトレースの
  詳細がログファイルへ出力されます。

* **GUI通知**:
  ``QApplication`` が起動していれば、画面上に「致命的なエラー」ダイアログが表示され、
  エラーメッセージおよび詳細（スタックトレース）を確認・コピーできます。

.. _opvjvl-logging-sigint:

6. Ctrl+C（SIGINT）による正常終了
------------------------------------

Qtのイベントループ実行中は、CPythonのシグナルハンドラがバイトコード命令の合間でしか
呼び出されないため、通常はターミナルでCtrl+Cを押してもすぐには終了しません。本ソフト
ウェアでは、短い間隔（既定200ms）で空処理を実行する ``QTimer`` を走らせておくことで
定期的にPythonインタプリタへ制御を戻し、シグナルが確実に配送されるようにしています
（``app.py`` の ``install_sigint_handler``）。

SIGINTハンドラは ``window.close()`` を呼ぶだけにとどめてあり、これにより
``MainWindow.closeEvent``（実行中測定への協調的な中断要求、機器出力OFFの待機、
測定パラメータの ``settings.json`` への保存）が通常のウィンドウクローズと同じ経路で
必ず実行されます。
